# BoardFlowBench 中文说明与扩展指南

本文档用于快速理解 `BoardFlowBench` 文件夹的作用、当前项目结构、后续可扩展的函数功能，以及可以沉淀成 Codex skill 的方向。

## 1. 项目定位

`BoardFlowBench` 是一个用于评估多轮 coding agent 交接能力的实验型 benchmark。

它关注的问题不是某个业务功能本身有多复杂，而是：

- agent 是否会先阅读仓库内的共享状态文件；
- agent 是否能理解当前任务、依赖关系和允许修改范围；
- agent 是否会把进度写回项目看板；
- agent 是否会在交接时留下结构化 handoff；
- 下一个 agent 是否能只依赖仓库状态继续工作，而不是依赖上一轮聊天记录。

简单理解：它把“多人或多 agent 协作时的信息交接”做成一个可以测试、评分和对比的实验环境。

## 2. 核心目录结构

```text
BoardFlowBench/
├── AGENTS.md                 # agent 进入项目后必须先读的入口说明
├── AI_CONTRACT.md            # 稳定规则：任务纪律、文件卫生、完成标准
├── PROJECT_BOARD.md          # 人类可读的项目看板
├── .board/
│   ├── tasks.yaml            # 机器可读的任务看板
│   ├── handoff.schema.json   # handoff JSON 的结构约束
│   └── handoffs/             # 每次交接留下的结构化记录
├── benchmark/
│   ├── targets/              # 独立 demo 仓库地址与固定 seed commit
│   ├── tasks/expense_lite/   # 每个 benchmark 任务的定义和验收标准
│   └── scenarios/            # 实验条件，例如有无 BoardFlow 协议
├── project/tasks/            # BoardFlowBench 自身研发任务
├── docs/                     # 设计、实验流程、研究计划等说明文档
└── scripts/                  # 后续可放实验运行脚本
```

## 3. BoardFlow 工作流

在 `full_boardflow` 场景中，一个 agent 开始任务时应按下面顺序读取文件。`boardflow_sequential` 作为兼容别名保留：

1. `AGENTS.md`
2. `AI_CONTRACT.md`
3. `PROJECT_BOARD.md`
4. `.board/tasks.yaml`
5. 当前 workspace 中的 `.board/assigned_task.yaml`
6. 最近相关的 `.board/handoffs/*.json`

读取这些文件后，agent 应确认：

- 当前 milestone 是什么；
- 被分配的 task id 是什么；
- 依赖任务是否已经完成；
- 哪些文件允许修改；
- 需要运行哪些验收命令；
- 停止工作前是否需要写 handoff。

这个流程的重点是让仓库本身成为协作状态的来源，而不是让下一位 agent 依赖上一轮对话。

## 4. 当前 demo 任务链

Expense Lite 是一个独立 Git 仓库中的极简 expense tracker。每次实验从固定 seed commit clone 临时 workspace，用来承载连续任务。BoardFlowBench 根仓库不直接完善 demo 功能。

当前任务链如下：

| 任务 | 目标 | 主要文件 |
| --- | --- | --- |
| B001 | 日期解析 | `src/expense_lite/parser.py` |
| B002 | CSV 导入 | `parser.py`, `validator.py` |
| B003 | 月度汇总 | `summary.py` |
| B004 | Markdown 报告 | `report.py`, `cli.py` |

这个 demo 故意保持简单，因为 benchmark 真正评估的是 agent 的交接纪律、范围控制、测试记录和状态同步能力。

## 5. Benchmark Target 维护

Expense Lite 是测试平台，不是 BoardFlowBench 的产品路线。需要调整 benchmark 难度时，在独立 demo 仓库中维护 seed、reference-history 和私有 oracle，再同步更新本仓库的 target manifest、task YAML 和 scenario runner。不要在 BoardFlowBench 根目录中加入 demo 实现。

### 5.1 常用术语

- `seed`：实验开始前 clone 的固定起始 commit。它保留基础平台和刻意留下的任务缺口，不包含参考答案。
- `reference-history`：独立 demo 仓库中保存完整参考实现的分支。它不作为 seed 的祖先，避免 agent 通过 `git log` 找到答案。
- `scope`：一个 sticker 被允许修改的文件边界。scorer 会比较 sticker 起始 baseline commit 与当前 workspace，检查是否修改了 `allowed_paths` 以外的文件。
- `oracle`：放在 agent workspace 之外的私有验收测试。agent 不能读取 oracle，但 finalize gate 会运行它。
- `handoff`：agent 结束一段工作时留下的结构化交接记录。空 handoff 不应被接受。
- `sticker`：blackboard 上的一张任务卡，对应一个 B 系列或 P 系列 task。

## 6. 可以扩展的 benchmark/scoring 功能

`tools/benchmark_scorer.py` 和 `repo_manager_core/board/` 负责自动评分逻辑。

可新增或增强的检查函数：

```python
def check_acceptance_commands(repo: str, task: dict) -> dict:
    """运行任务定义中的验收命令，并记录通过/失败结果。"""

def check_handoff_schema(repo: str, task: dict) -> dict:
    """检查 handoff JSON 是否符合 handoff.schema.json。"""

def check_board_sync(repo: str) -> dict:
    """检查 PROJECT_BOARD.md 与 .board/tasks.yaml 的状态是否一致。"""

def check_scope_violations(repo: str, task: dict) -> dict:
    """检查 git diff 是否超出 allowed_paths。"""

def check_temp_file_hygiene(repo: str) -> dict:
    """检查是否产生了未记录的临时文件、缓存文件或根目录垃圾文件。"""
```

这些检查已经接入 `tools/benchmark_scorer.py`，输出机器可读的 JSON 结果。

## 7. 可以沉淀成 Codex skill 的方向

如果你后续希望让 Codex 更稳定地处理这个项目，可以把下面流程做成一个专用 skill。

### 7.1 BoardFlow Agent Skill

用途：

- 开始任务时自动按 BoardFlow 读取顺序检查文件；
- 自动提取 task id、allowed paths、acceptance commands；
- 工作结束时提醒更新 board 和 handoff；
- 避免修改未授权文件。

建议 skill 名称：

```text
boardflow-agent
```

核心说明可以包括：

- 必须读取 `AGENTS.md`;
- 必须读取 `.board/tasks.yaml`;
- 未指定 task id 时不能擅自选择任务；
- 开始真实开发前把任务状态改为 `IN_PROGRESS`;
- 停止前创建 `.board/handoffs/*.json`;
- 最终回复必须列出修改文件、测试命令和结果。

### 7.2 BoardFlow Reviewer Skill

用途：

- 专门做代码审查和实验协议审查；
- 检查 board 状态、handoff、scope、tests、hygiene；
- 输出问题清单，而不是直接修代码。

当前 skill 名称：

```text
boardflow-reviewer
```

核心检查项：

- `PROJECT_BOARD.md` 和 `.board/tasks.yaml` 是否一致；
- task 状态是否只使用合法值；
- 是否修改了 allowed paths 之外的文件；
- handoff 是否符合 schema；
- 测试是否运行，结果是否被记录；
- 是否有未清理或未记录的临时文件。

### 7.3 Expense Lite Implementer Skill

用途：

- 专门处理独立 Expense Lite workspace 中 `src/expense_lite/` 内的功能任务；
- 维护 parser、validator、summary、report、cli 的职责边界；
- 写功能时同步补测试。

建议 skill 名称：

```text
expense-lite-implementer
```

核心约束：

- 日期解析逻辑集中在 `parser.normalize_date`;
- 校验逻辑集中在 `validator`;
- 汇总逻辑集中在 `summary`;
- Markdown 输出集中在 `report`;
- CLI 不写复杂业务逻辑，只编排模块调用。

## 8. 推荐的下一步开发顺序

建议按照下面顺序推进，避免项目过早变复杂：

1. 维护独立 demo target 的固定 seed commit。
2. 增强 scorer 的自动检查和 calibration fixtures。
3. 维护 scenario runner，自动初始化 no-board、native instructions、native docs handoff 和 full BoardFlow 四种条件。
4. 设计结果聚合表，统计 context loss、scope violation、handoff completeness 等指标。
5. 把稳定流程沉淀成 `boardflow-agent` 和 `boardflow-reviewer` skills。

## 9. 写新任务时的模板

新增任务时，建议同时创建或更新：

```text
benchmark/tasks/expense_lite/bXXX_name.yaml
tests/test_xxx.py
src/expense_lite/xxx.py
```

任务 YAML 应至少包含：

```yaml
task_id: BXXX
title: Short title
intent: What this task adds
allowed_paths:
  - src/expense_lite/target.py
  - tests/test_target.py
acceptance:
  - Expected behavior 1
  - Expected behavior 2
acceptance_commands:
  - PYTHONDONTWRITEBYTECODE=1 PYTHONPATH=src python3 -m unittest discover -s tests -p 'test_target.py'
```

## 10. 注意事项

- 不要把动态任务状态写进 `AI_CONTRACT.md`。
- 不要在没有 task id 的情况下擅自开始某个 benchmark 任务。
- 不要提前完成未来 B 系列任务，否则 benchmark 无法区分每一步交接效果。
- 不要把临时文件放在仓库根目录。
- 修改 task 状态时，`PROJECT_BOARD.md` 和 `.board/tasks.yaml` 要同步。
- 如果没有运行测试，要在 handoff 中明确记录，而不是假装通过。
